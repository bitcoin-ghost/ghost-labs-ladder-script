Fee-Gated Covenant

A covenant that restricts spending based on fee rate and output amount. The SEND path requires a valid signature, the transaction fee rate to be within 5–50 sat/vB (HYSTERESIS_FEE), and the output to be within 10,000–100,000 sats (AMOUNT_LOCK). This prevents panic-spending during fee spikes and enforces minimum output sizes. An EMERGENCY path bypasses all constraints with dual signatures.

HYSTERESIS_FEE · 5–50 sat/vB AMOUNT_LOCK · 10k–100k sats SCHNORR · SCHEME 0x01
Rung Structure

Two outputs with separate spending conditions. The SEND output (vout 0) enforces fee and amount constraints for normal operation. The EMERGENCY output (vout 1) requires two signatures but has no other restrictions.

HYSTERESIS_FEE checks the transaction's fee rate (total fee / weight in sat/vB) falls within the band [5, 50]. AMOUNT_LOCK checks the output value is between 10,000 and 100,000 sats. Both are TX-structure constraints — evaluated from the transaction itself, requiring no witness data.

How Fee-Gated Covenants Work
1.
The funding transaction creates two outputs. Vout 0 (SEND) contains three blocks: SIG for authentication, HYSTERESIS_FEE constraining the fee rate to 5–50 sat/vB, and AMOUNT_LOCK constraining the output to 10,000–100,000 sats. Vout 1 (EMERGENCY) requires two signatures with no constraints.
2.
To spend via the SEND path, the owner signs the transaction. The evaluator computes fee_rate = total_fee / tx_weight_vB and checks it falls within the HYSTERESIS_FEE band. It also checks the output amount is within the AMOUNT_LOCK range. Both are introspection constraints — no witness data needed.
3.
If fees spike above 50 sat/vB (e.g. during congestion), the SEND path is blocked — preventing panic-spending at inflated rates. The AMOUNT_LOCK prevents dust outputs and ensures meaningful value transfer.
4.
The EMERGENCY path on vout 1 bypasses all constraints: the owner + cosigner can spend regardless of fee rate or amount. This is the escape hatch for genuine emergencies when fee conditions cannot be met.
